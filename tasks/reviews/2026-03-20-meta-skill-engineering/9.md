# Meta-Skill-Engineering — full repository review (latest attached snapshot)

## Executive summary

This repository is now **substantially stronger** than the earlier versions I reviewed. It has a leaner active inventory, a clearer internal-only contract, a seeded corpus, a more capable evaluation runner, and much better separation between active and archived concerns.

My overall judgment on this snapshot is:

- **Skill-library quality:** **good**
- **Single-skill authoring / testing system:** **good but not finished**
- **Full meta-skill evaluation system:** **promising, but still partial**
- **Packaged repo as-delivered:** **not fully runnable on Unix without cleanup**

### Bottom-line ratings

1. **Against official Agent Skills authoring/testing best practices:** **7.7 / 10**
2. **As a complete meta-skill benchmarking / improvement system:** **5.8 / 10**

The first score is higher because the repository now aligns well with the core Agent Skills ideas: lightweight `SKILL.md` packages, progressive disclosure, strong descriptions, real trigger testing, moderate length, and optional scripts/references where they add value. The second score is lower because the repository still does **not** provide a fully closed-loop, fully automated, two-layer meta-skill evaluation system with real downstream execution benchmarking and executable regression coverage.

For actual script inspection and limited execution, I used a temporary normalized copy so I could verify logic. Where I did that, I state it explicitly.

## External best-practice yardstick used for this review

I compared the repository against the official Agent Skills documentation the user specified:

- What are skills — folder with required `SKILL.md`, optional scripts/references/assets, progressive disclosure, required frontmatter limited to `name` and `description`, and no fixed required body structure at the spec level.  
  Source: https://agentskills.io/what-are-skills
- Quickstart — `description` determines activation; skills are lightweight folders; name should match folder.  
  Source: https://agentskills.io/skill-creation/quickstart
- Best practices — start from real expertise, keep coherent units, aim for moderate detail, keep `SKILL.md` under ~500 lines / 5000 tokens, use progressive disclosure, provide defaults, and favor procedures over declarations.  
  Source: https://agentskills.io/skill-creation/best-practices
- Optimizing descriptions — `description` is the primary trigger mechanism; use realistic positive/negative queries, logs/tool calls where possible, and train/validation splits to avoid overfitting.  
  Source: https://agentskills.io/skill-creation/optimizing-descriptions
- Evaluating skills — use realistic prompts, expected outputs, assertions, and compare with/without the skill; official examples still use `evals/evals.json` as a simple reference format.  
  Source: https://agentskills.io/skill-creation/evaluating-skills
- Using scripts — scripts are optional, commands are relative to the skill directory root, avoid interactive prompts, document usage, and prefer structured output.  
  Source: https://agentskills.io/skill-creation/using-scripts

This review also uses the user’s own higher-bar meta-skill evaluation notes as a secondary benchmark for what a **serious** meta-skill evaluation system would eventually need: artifact benchmark + downstream execution benchmark + regression benchmark, richer target corpus categories, stronger release gates, and score-history discipline.  
Source file: `Evaluating Meta-Skills.txt` (uploaded project source).

## What is materially improved in this snapshot

## 1. The repo contract is much clearer

`AGENTS.md` is now the real contract for the active repository. It defines:

- internal-only usage
- canonical root directories
- canonical eval file names
- canonical `SKILL.md` section order
- a hard rejection of stale active eval formats like `evals.json`, `output-tests.jsonl`, `triggers.yaml`, `outputs.yaml`, and `baselines.yaml`

Evidence: `AGENTS.md:33-78`, especially `AGENTS.md:39-65`.

### Why this is good

This brings the repo much closer to a real internal platform rather than a loose pile of prompt files.

It also aligns well with the official Agent Skills model that skills are lightweight folders with `SKILL.md` plus optional support layers, and that the main activation logic lives in `name` + `description`. The repo is stricter than the spec on body structure, but that is acceptable as an internal convention.  
Official basis: Agent Skills “What are skills?” and “Quickstart”.

## 2. The active inventory is leaner and more coherent

The repository now focuses the active surface on 12 skills and moves distribution/import/provenance tooling to archive. That is appropriate for an internal-only system.

Evidence: `README.md:7-14`, `README.md:44-61`, `archive/README.md`.

### Why this is good

For an internal library, a smaller active surface lowers routing overlap, maintenance overhead, and contract drift.

## 3. `skill-creator` is much improved

The live `skill-creator` no longer pushes the old active `evals/evals.json` pattern into the repo’s core contract. Instead, it points toward the canonical split eval files and the current test/review/improve flow.  
Evidence: `skill-creator/SKILL.md:188-194`.

### Why this is good

This removes one of the worst sources of self-reintroduced schema drift from earlier versions.

## 4. `skill-lifecycle-management` is improved

Earlier versions tried to store lifecycle state in frontmatter (`metadata.maturity`), which directly conflicted with the repo’s internal-only frontmatter rules. The latest version instead refers to a lifecycle index and in-body notices rather than polluting frontmatter.  
Evidence: `skill-lifecycle-management/SKILL.md:56`, `skill-lifecycle-management/SKILL.md:109`, `skill-lifecycle-management/SKILL.md:169`.

### Why this is good

This is much more compatible with the internal contract that frontmatter should remain minimal.

## 5. The evaluation runner is meaningfully stronger

`run-evals.sh` is no longer just a name-grep trigger proxy. It now has:

- `--observe` mode, which parses structured JSON output and tries to detect actual `SKILL.md` reads
- `--strict` mode, which performs a differential run with and without custom instructions
- `--runs N` majority-vote execution
- optional usefulness scoring (`--usefulness`) as an LLM judge layer

Evidence: `scripts/run-evals.sh:27-32`, `scripts/run-evals.sh:83-89`, `scripts/run-evals.sh:392-393`, `scripts/run-evals.sh:916`, `AGENTS.md:126-137`, `docs/evaluation-cadence.md:57-72`.

### Why this is good

This aligns much better with the official guidance for optimizing descriptions, which recommends using client logs/tool-use evidence where possible rather than naive textual proxies, and recommends repeated runs plus train/validation splits when optimizing triggering.  
Official basis: Agent Skills “Optimizing descriptions”.

## 6. Trigger optimization is now much closer to official best practice

`run-trigger-optimization.sh` uses a 60/40 train/test split and repeated evaluation.  
Evidence: `scripts/run-trigger-optimization.sh:97`, `scripts/run-trigger-optimization.sh:428-474`, `AGENTS.md:139-141`.

### Why this is good

That closely matches the official recommendation to avoid overfitting by separating train queries from validation queries and checking whether improvements generalize.  
Official basis: Agent Skills “Optimizing descriptions”.

## 7. Script distribution is cleaner

The repo now treats root `scripts/` as the source of truth and deploys copies into skill-local `scripts/` directories using `sync-to-skills.sh`. `AGENTS.md` documents this.  
Evidence: `AGENTS.md:17-31`.

### Why this is good

It keeps operational code centralized while still making script-backed skills self-contained at the package level.

## Detailed review — scripts and operational tooling

## Cross-cutting operational defect: CRLF line endings in shipped scripts

### Evidence

The attached zip snapshot preserves CRLF line endings in the shell scripts. Running the root validator in the extracted repo fails immediately on Unix with:

```text
scripts/validate-skills.sh: line 2: set: pipefail\r: invalid option name
```

The repository’s `.gitattributes` currently contains only:

```gitattributes
* text=auto
```

Evidence: `.gitattributes:1-2`.

### Why this matters

This is a real packaging/delivery defect. A repo that advertises validation and evaluation flows should be runnable in its intended environment without manual line-ending repair.

### Recommendation

- force LF for `*.sh`, `*.py`, `*.md`, `*.jsonl`, and `*.mjs`
- ensure executable bits are preserved for runnable shell scripts
- add a quick line-ending check to validation or CI

## Review of the evaluation system against best practices

## What the system now does well

### 1. It treats descriptions as a real routing surface

This is correct and aligned with the official docs. The repository now treats the `description` field as routing logic and has dedicated tooling to test and optimize it.  
Evidence: `skill-trigger-optimization/SKILL.md`, `scripts/run-trigger-optimization.sh`, `AGENTS.md:137-141`.

### 2. It uses observation/logging for routing evaluation

This is much closer to best practice than name-grepping.  
Evidence: `scripts/run-evals.sh:27-32`, official optimizing-descriptions guidance.

### 3. It uses repeated runs and train/test splits where they matter most

This is strong and directly aligned with official best practice for description optimization.  
Evidence: `scripts/run-trigger-optimization.sh:97`, `docs/evaluation-cadence.md:71-72`.

### 4. It now has a seeded corpus

Weak / strong / adversarial / regression is the right category family for a meta-skill system.

### 5. It includes a second-layer usefulness signal

The optional usefulness judge is a useful step beyond simple pattern checks.

## What the system still does poorly

### 1. It is not yet a full two-layer automated meta-skill evaluator

The corpus path is still:

- Layer 1 automated structural/precheck
- Layer 2 manual comparison / invocation

That is not yet the full target architecture described in the user’s own evaluation notes.

### 2. It does not yet provide a true automated no-skill baseline for skill-evaluation

`skill-evaluation` still claims more than the runner actually does.

### 3. The regression loop is incomplete

Failures can be harvested, but not all harvested failures become machine-runnable regressions.

### 4. The corpus Layer 1 metric is too structural for weak-skills testing

Because `run-corpus-eval.sh` currently uses structure-heavy prechecks, several weak skills still score as “valid” in Layer 1 even though their real weakness is semantic / procedural rather than structural.

That means corpus evaluation is currently better at detecting malformed skills than at proving meta-skill improvement quality.
