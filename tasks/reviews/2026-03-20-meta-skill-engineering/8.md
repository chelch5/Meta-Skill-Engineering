# Meta-Skill Engineering — Comprehensive Audit Report

**Prepared:** 20 March 2026  
**Repository:** `chelch5/Meta-Skill-Engineering` (cloned from GitHub)  
**Reference standard:** [agentskills.io](https://agentskills.io) documentation suite  
**Comparator:** `/mnt/skills/examples/skill-creator` (reference Anthropic skill)

---

## Executive Summary

The Meta-Skill Engineering (MSE) repository is a well-conceived, coherently structured suite of 12 skills designed for the full lifecycle of agent skill management. The architecture is sound, the pipeline documentation is accurate and consistent across `README.md` and `AGENTS.md`, and all skills conform to the required section structure. The evaluation system is sophisticated — it implements multi-run majority voting, LLM-as-Judge usefulness scoring, train/test splits for trigger optimisation, and a structural gate system.

However, the audit identified **two systemic technical issues**, **one significant automation gap**, and **multiple stale data references** that collectively undermine the reliability of the evaluation system. These are fixable, but they represent a meaningful quality regression relative to the system's stated goals.

## Section 3: Evaluation System Analysis

This is the most important section of the audit. The evaluation system is sophisticated but contains issues that reduce its reliability.

### 3.1 [CRITICAL] Precision/Recall Terminology Is Inverted in `run-evals.sh`

**Location:** `scripts/run-evals.sh`, lines 767–786 and 853–854.

In information retrieval, the standard definitions are:
- **Precision** = TP / (TP + FP) — of things the system said were relevant, how many actually were?
- **Recall** = TP / (TP + FN) — of things that actually were relevant, how many did the system find?

The script calculates:
```bash
precision=$((GATE_POS_PASS * 100 / GATE_POS_TOTAL))   # positive trigger pass rate
recall=$((GATE_NEG_PASS * 100 / GATE_NEG_TOTAL))       # negative trigger pass rate
```

These are inverted. `GATE_POS_PASS / GATE_POS_TOTAL` (the fraction of positive cases where the skill correctly triggered) is **Recall** — it measures what percentage of genuinely-relevant prompts the skill detected. `GATE_NEG_PASS / GATE_NEG_TOTAL` (the fraction of negative cases where the skill correctly stayed silent) is **Specificity** — not Recall at all.

This is confirmed by `skill-evaluation/SKILL.md` line 103, which correctly defines these as:
```
Precision = TP / (TP + FP), Recall = TP / (TP + FN)
```
But the script cannot implement this correctly because it doesn't track false positives separately — it only tracks positive pass rate and negative pass rate.

The gate output reports:
```
| Trigger precision ≥ 80% | PASS | 90% |
| Trigger recall ≥ 80%    | PASS | 88% |
```
When what it is actually measuring is:
```
| Positive trigger pass rate ≥ 80% | PASS | 90% |   (this is recall, not precision)
| Negative trigger pass rate ≥ 80% | PASS | 88% |   (this is specificity, not recall)
```

**Functional impact:** The gates themselves are measuring reasonable things — "does the skill trigger on positive cases?" and "does the skill stay silent on negative cases?" — so results are not meaningless. But the labels on the output report are wrong, which means:
1. Anyone reading `eval-results/` reports will misinterpret the metrics.
2. `skill-evaluation/SKILL.md` Step 6 says "Precision ≥ 95% and recall ≥ 90%" but these are the script's positive/negative pass rates, not true precision/recall.
3. The metrics cannot be compared to standard IR benchmarks.

**Fix:** Either rename the variables and gate labels to `positive_pass_rate` / `negative_pass_rate` (simpler), or restructure the JSONL runner to track TP/FP/FN/TN counts properly (more correct but more complex). The former is a one-hour fix; the latter would require modest refactoring.

### 3.2 [HIGH] Threshold Inconsistency Between Documentation and Enforcement

`skill-evaluation/SKILL.md` Step 6 states:
> Routing target: precision ≥ 95% and recall ≥ 90%

And the output contract template shows `≥ 95%` and `≥ 90%` explicitly. But `run-evals.sh` enforces:
```bash
[[ $precision -ge 80 ]] && precision_status="PASS"
[[ $recall    -ge 80 ]] && recall_status="PASS"
```

A skill scoring 85% on both metrics would be `PASS` per the script but `Fail` per the documented targets. This means a human following `skill-evaluation`'s procedure would produce different pass/fail verdicts than the automated runner — the two systems disagree on what "passing" means.

Additionally, the `run-baseline-comparison.sh` script passes zero threshold parameters to `run-evals.sh`, so it also uses the 80% gates. A skill that "passes" the automated comparison may still fall short of the documented quality bar.

**Fix:** Reconcile the thresholds. Either update `skill-evaluation/SKILL.md` to use 80%/80% targets, or update `run-evals.sh` to use 95%/90% gates. The latter is the appropriate change if quality is the goal.

### 3.3 [HIGH] Stale References to Archived Skills in Eval Files

Seven lines across six eval files reference skills that have been moved to `archive/` and are no longer active:

| File | Line | Stale Reference |
|---|---|---|
| `skill-creator/evals/trigger-negative.jsonl` | 4 | `community-skill-harvester` |
| `skill-creator/evals/trigger-negative.jsonl` | 8 | `skill-packaging` |
| `skill-adaptation/evals/trigger-negative.jsonl` | 6 | `skill-packaging` |
| `skill-catalog-curation/evals/trigger-negative.jsonl` | 4 | `skill-packaging` |
| `skill-catalog-curation/evals/trigger-negative.jsonl` | 5 | `skill-installer` |
| `skill-lifecycle-management/evals/trigger-negative.jsonl` | 4 | `skill-packaging` |
| `skill-safety-review/evals/trigger-negative.jsonl` | 4 | `skill-provenance` |

These entries use the `"better_skill"` field to name the correct alternative skill for a given prompt. Since those archived skills no longer exist, when a model encounters the prompt *"Package this skill for distribution"* it has no correct routing target. This makes those specific negative test cases misleading — they may result in false positives if the model routes to an active skill (e.g., `skill-lifecycle-management`) rather than the archived `skill-packaging`, or false negatives if it routes nowhere.

The `validate-skills.sh` script does check for cross-references in SKILL.md body text pointing to non-existent skill directories, but it does not check `better_skill` fields in JSONL eval files. This gap means the stale references were not caught by the validator.

**Fix:** Update `better_skill` values to either `null` (no current equivalent) or the closest active skill. For `skill-packaging`, the closest equivalent is likely `skill-lifecycle-management`. For `community-skill-harvester`, no current equivalent exists — set to `null`. Additionally, extend `validate-skills.sh` to scan `better_skill` fields in JSONL files.

### 3.4 [MEDIUM] Layer 2 of Corpus Evaluation is Not Automated

`scripts/run-corpus-eval.sh` is documented as performing two layers:
- **Layer 1:** Did the meta-skill produce valid output? (structural checks)
- **Layer 2:** Does the rewritten skill perform better? (eval comparison)

Layer 1 is fully automated — it runs `check_skill_structure.py` against corpus skills and reports pre-scores. Layer 2 is not automated. The script literally contains:

```bash
# The actual meta-skill invocation requires copilot CLI (interactive).
# We prepare the harness: baseline is recorded, post-invocation comparison
# can be run via run-baseline-comparison.sh.
#
# To complete the eval loop manually:
#   1. Run the meta-skill on ${WORK_DIR}/working.md
#   2. Run: ./scripts/run-baseline-comparison.sh ...
```

This is a critical gap because Layer 2 is the actual test of whether the meta-skills work — Layer 1 only tells you if the *corpus skills themselves* are structurally valid before any meta-skill touches them.

The gap is acknowledged in code comments, but the documentation in `README.md`, `AGENTS.md`, and `docs/evaluation-cadence.md` does not clearly communicate that corpus evaluation is only partially automated. A developer running `run-full-cycle.sh` would get corpus eval output that only covers Layer 1, without any indication that the more important Layer 2 assessment requires manual intervention.

**Mitigation noted:** The limitation exists because Copilot CLI's meta-skill invocation requires interactive context. A non-interactive workaround could use `copilot -p "improve this skill" < corpus/weak/bad-triggers.md` but this would bypass the skill-triggering mechanism (which requires the skill to be in context). This is a genuine design constraint, not just a laziness gap.

**Fix:** At minimum, update `docs/evaluation-cadence.md` to prominently note that `run-corpus-eval.sh` only performs Layer 1 (structural pre-scoring). Provide a separate manual runbook for Layer 2. Consider whether a non-interactive approximation is feasible.

### 3.5 [MEDIUM] `quick_validate.py` Not in Sync Manifest

`scripts/quick_validate.py` exists in the root `scripts/` directory but is not included in the manifest in `sync-to-skills.sh`. The manifest determines which scripts are distributed to per-skill `scripts/` directories.

```bash
# From sync-to-skills.sh MANIFEST (showing what IS included):
"run-evals.sh:skill-evaluation,skill-benchmarking,skill-creator,skill-testing-harness"
"run-baseline-comparison.sh:skill-improver,skill-benchmarking"
"check_skill_structure.py:skill-evaluation,skill-anti-patterns,skill-safety-review,skill-creator"
# ... but no quick_validate.py entry
```

The implication is that `quick_validate.py` is available at the repo root but no skill can reference it with a relative `scripts/quick_validate.py` path. If any skill procedure ever references this script, it would be a phantom reference. Currently none do, so this is a low-severity issue — but it represents a divergence between what exists in `scripts/` and what the sync mechanism distributes.

**Fix:** Either add `quick_validate.py` to the manifest with the skills that would use it, or remove it from `scripts/` if it is no longer needed.

### 3.6 [LOW] Behaviour Test Coverage Is at the Minimum

Every skill has exactly 3 behaviour tests in `behavior.jsonl` (except `skill-improver` which has 4). AGENTS.md sets the minimum at 1, and the system appears to have targeted exactly 3 for consistency.

agentskills.io's guidance says *"Start with 2–3 test cases. Don't over-invest before you've seen your first round of results. You can expand the set later."* So 3 is a reasonable starting point. However, `skill-creator` and `skill-improver` are the two most complex skills in the system and are the most likely to exhibit edge-case failures in output quality. They would benefit from expanded behaviour coverage (6–8 cases each) to test the full phase-by-phase procedure rather than just surface-level structural compliance.

**Additional note:** Only `skill-creator`'s behavior tests (2/3 cases) have `usefulness_criteria` defined, enabling LLM-as-Judge scoring. All other skills have zero `usefulness_criteria` fields, making `--usefulness` mode effectively a no-op for 11 of 12 skills. This significantly limits the usefulness of the fifth gate. Given the sophistication of the judge infrastructure (multi-run median voting, per-dimension scoring), this is an under-utilised capability.

### 3.7 [LOW] Description Extraction in `run-trigger-optimization.sh` Is Fragile

The script extracts the current description with a multi-stage `sed` pipeline (line ~148):
```bash
CURRENT_DESC=$(sed -n '/^---$/,/^---$/p' "$SKILL_MD" | \
  grep -A 100 'description:' | ... | sed 's/description: *>- *//' | xargs)
```

This sed chain does not correctly handle all valid YAML multi-line scalar formats. It assumes `>-` folded block scalar format. All current descriptions use `>-`, so this works today. But if any description is ever written as a plain scalar or quoted string, the extraction would fail silently — the script would patch the wrong text into the frontmatter.

The Python-based YAML manipulation (also in the same script) uses `re.sub` rather than a proper YAML parser, which has similar fragility. The `python3 -c` block that patches the frontmatter concatenates strings with inline regex substitution — this could mangle the description if it contains regex special characters.

**Fix:** Use `python3 -c "import yaml; ..."` for both extraction and patching. Since the project already uses Python 3 for `check_skill_structure.py`, adding `import yaml` to the trigger-optimization script is low-friction.

## Section 4: Script Architecture Assessment

### 4.1 `run-evals.sh` (509 lines)

This is the most important script and is well-implemented overall. Notable strengths:

- **Observe mode** parses the Copilot CLI JSON output stream to detect actual `SKILL.md` file reads via the `view` tool. This is the correct approach — detecting whether the skill was *loaded*, not whether the model mentioned it.
- **Strict mode** uses differential testing (with vs without `--no-custom-instructions`) as a more rigorous but slower alternative. Both modes are appropriate for different situations.
- **Multi-run majority voting** (`--runs N`) is correctly implemented with `MAJORITY_THRESHOLD=$(( (RUNS + 1) / 2 ))`. This is the right formula: at N=3, threshold=2; at N=5, threshold=3.
- **LLM-as-Judge** with per-dimension scoring (correctness, completeness, actionability, conciseness) and median aggregation across multiple judge runs is well-designed. The prompt correctly requests raw JSON without markdown fences and handles malformed responses gracefully.
- **JSON output mode** (`--json`) enables machine-readable summaries for consumption by `run-baseline-comparison.sh`.

**Issue noted above in 3.1:** The precision/recall variable naming is inverted.

### 4.2 `run-baseline-comparison.sh` (230 lines)

Well-implemented. The five quality gates (section count, no section deletions, line count < 500, name preserved, eval regression) are the correct post-modification checks. The gate output format is clear and parseable.

**Strength:** The eval regression gate (Gate 5) creates a temporary skill directory, copies eval files from the original skill, swaps in the modified SKILL.md, runs evals, swaps back, runs evals again, and compares. This is a correct implementation of the regression check described in agentskills.io's guidance on iterating on skills.

### 4.3 `run-trigger-optimization.sh` (260 lines)

Implements the agentskills.io optimization loop faithfully: 60/40 train/test split, baseline evaluation, LLM-proposed improvement, train set re-evaluation, held-out test set validation, and ACCEPT/REJECT verdict. The script does not auto-apply changes — it requires manual review. This is the correct behaviour.

**Note on split algorithm:** The interleaving split (`i % 5 == 3 || i % 5 == 0`) correctly produces approximately 40% test cases, preserving category diversity better than a simple head/tail split.

**Fragility noted above in 3.7:** Description extraction and frontmatter patching use fragile sed/regex pipelines.

### 4.4 `validate-skills.sh` (130 lines)

Comprehensive structural validator. Checks: frontmatter fields, unexpected frontmatter fields, canonical section headings, stale `manifest.yaml` files, line count, cross-reference integrity (pointing to existing skill directories), phantom file references (scripts/references files mentioned but not present), and JSONL format validity.

**Gap identified:** Does not check `better_skill` fields in JSONL eval files against the active skill inventory. This allowed the seven stale archived-skill references (Section 3.3) to persist undetected.

### 4.5 `sync-to-skills.sh`

Well-implemented with `--dry-run` and `--check` modes. The manifest-driven approach (source-of-truth in root `scripts/`, deployed copies in per-skill directories) is a good architectural choice that prevents script drift.

**Gap:** `quick_validate.py` is absent from the manifest (Section 3.5).

### 4.6 `run-corpus-eval.sh`

Layer 1 (structural pre-scoring) is well-implemented. Layer 2 is not automated (Section 3.4). The script correctly distinguishes behaviour expectations by corpus tier (strong tier = gate fails on invalid; weak/adversarial tier = issues are expected).

### 4.7 Python Support Scripts

`check_skill_structure.py` is clean and well-documented. The 10-point scoring system is calibrated sensibly: frontmatter (3 points), required headings (6 points), heading order (1 point). Line count and phantom references are warnings only (not scored), which is the right call since they don't indicate structural failure.

`skill_lint.py` is a simpler, faster alternative focused on actionable code-style checks. The routing cue detection (scanning for `use when`, `when the user`, `when someone`) is a practical heuristic.

`check_preservation.py` focuses on word-level preservation of key sections during edits — this supports the "preserve core purpose" requirement in `skill-improver`.

## Section 5: Pipeline Flows and Inter-Skill Routing

### 5.1 Creation Pipeline Analysis

```
skill-creator → skill-testing-harness → skill-evaluation
    → skill-trigger-optimization → skill-safety-review → skill-lifecycle-management
```

This pipeline is correctly documented and the cross-references between skills are consistent. Each skill's "Next steps" section points to the correct downstream skill. The only issue is the ordering ambiguity in `skill-creator`'s finalization prose (Section 2.3).

### 5.2 Improvement Pipeline Analysis

```
skill-evaluation → skill-anti-patterns → skill-improver → skill-trigger-optimization
```

The handoff mechanism between `skill-evaluation` and `skill-improver` (via structured `Handoff` section in eval reports written to `eval-results/<skill>-eval.md`) is well-designed. `skill-improver`'s Phase 1 explicitly instructs reading this file, and Phase 2 uses the eval signals as primary diagnosis input.

**Strength:** The eval-driven diagnosis table in `skill-improver` maps specific eval signals to specific failure modes and fix targets. This is the kind of concrete, actionable guidance that elevates the skill above generic advice.

### 5.3 Library Management Pipeline

```
skill-catalog-curation → skill-lifecycle-management
```

Simple and correct. Both skills have clear "When NOT to use" sections that prevent confusion with each other.

### 5.4 Negative Boundary Coverage

All 12 skills have `# When NOT to use` sections naming specific alternative skills with `→ skill-name` syntax. The cross-references form a coherent graph — when skill A says "not for X, use skill-B," skill-B is a real, active skill. No circular redirections detected.

One minor gap: `skill-catalog-curation` and `skill-lifecycle-management` have clear boundaries between themselves, but `skill-catalog-curation` does not mention `skill-variant-splitting` as an alternative for when the real problem is a single bloated skill (not a library-level issue). A user wanting to "clean up" a broad skill could incorrectly invoke catalog curation when variant splitting is the correct tool.

## Section 7: Comparison to Reference Skill-Creator

The reference skill at `/mnt/skills/examples/skill-creator/` (Anthropic's built-in implementation) differs from the MSE `skill-creator` in several meaningful ways.

### 7.1 Architecture

| Dimension | MSE `skill-creator` | Reference `skill-creator` |
|---|---|---|
| SKILL.md length | 328 lines | 485 lines |
| Procedure style | Structured phases (1–6) | Narrative conversation guide |
| Agent prompts | Embedded in SKILL.md | Separate `agents/` directory (analyzer.md, comparator.md, grader.md) |
| Eval format | JSONL (3 files) | `evals.json` (JSON array) |
| Automation | Bash scripts | Python scripts |
| HTML eval viewer | Not present | `eval-viewer/viewer.html` + `generate_review.py` |
| Scope | Creation + testing only | Creation + improvement + benchmarking (combined) |

### 7.2 Description Quality

The reference description is:
> "Create new skills, modify and improve existing skills, and measure skill performance. Use when users want to create a skill from scratch, edit, or optimize an existing skill, run evals to test a skill, benchmark skill performance with variance analysis, or optimize a skill's description for better triggering accuracy."

There is **no negative boundary** (*"Do not use for..."*). By agentskills.io's guidance and MSE's own `skill-creator/SKILL.md`, every description should end with "Do not use for..." naming adjacent skills. The reference description fails its own skill's rule.

The MSE `skill-creator` description correctly includes: *"Do not use for splitting a broad skill into variants (skill-variant-splitting), adapting a skill to a different environment (skill-adaptation), improving an existing skill without full iteration (skill-improver), or running standalone evaluations without creation intent (skill-evaluation)."*

### 7.3 Scope Comparison

The reference skill-creator combines creation, improvement, and benchmarking into a single skill. MSE correctly separates these into distinct skills with clear boundaries. The MSE approach is more aligned with agentskills.io's principle of *"coherent units"* — each skill should *"encapsulate a coherent unit of work that composes well with other skills."* A single skill covering creation + improvement + benchmarking is overloaded.

The reference's casual narrative style (*"Cool? Cool."*) is effective for Claude's conversational agent context but would likely underperform in a structured pipeline context like MSE, where precise routing and predictable output formats matter more.

### 7.4 Eval Format Incompatibility

The reference uses `evals.json` (JSON array with `id`, `prompt`, `expected_output`, `files`, `assertions` fields). MSE's AGENTS.md explicitly prohibits this format: *"Do not use `evals.json`."* This means the reference skill's eval infrastructure is incompatible with MSE's `run-evals.sh` runner. They are designed for different agent environments (Claude Code vs GitHub Copilot CLI) and the format divergence reflects this.

### 7.5 Tooling Sophistication

The reference has more sophisticated Python tooling: `run_eval.py` runs evaluations programmatically; `run_loop.py` implements the improvement loop; `generate_report.py` + `viewer.html` provide a live HTML dashboard for watching eval results in real time. This is more capable than MSE's bash-based approach for long evaluation runs.

However, MSE's bash scripts are simpler to understand, modify, and debug — a deliberate trade-off for a system whose users include AI agents.
